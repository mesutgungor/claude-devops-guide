How this works

Claude Code is a terminal-native AI agent with a bash tool, full filesystem access, and extensibility via MCP servers. This means it can read configs, write Terraform, run CLI commands, trigger pipelines, query monitoring data, and execute shell scripts — all from a single session. The guide below shows exactly how to configure and prompt Claude Code for each DevOps domain.

01 — Foundation Setup

Required First

Before Claude Code can own your DevOps, it needs three things: the CLI installed, cloud credentials available, and a CLAUDE.md project file describing your stack.

Install Claude Code

# Requires Node.js 18+
npm install -g @anthropic-ai/claude-code
# Navigate to your project
cd /path/to/project
# Initialize (generates CLAUDE.md)
claude /init

Expose Cloud Credentials

# AWS — use profiles, not inline keys
export AWS_PROFILE=production
export AWS_REGION=eu-west-1
# GCP
gcloud auth application-default login

az login
export ARM_SUBSCRIPTION_ID=xxxxx

    05 — Security Operations

    SecOps · SAST · Secrets
    Static analysis (SAST)

    Runs Semgrep, Bandit (Python), ESLint security rules, or Snyk CLI against the codebase. Interprets findings and patches them.

    semgrep --config=auto src/ --json
    snyk code test --json
    # Then: "Fix all HIGH severity findings"
    Secret scanning

    Uses gitleaks or trufflehog to detect exposed credentials in git history and working directory. Rotates secrets via CLI if found.

    gitleaks detect --source . --report-path leak-report.json
    trufflehog git file://. --json
    Dependency scanning

    Audits npm/pip/maven dependencies for CVEs, generates SBOM, upgrades vulnerable packages.

    npm audit --json
    pip-audit --output json
    snyk test --all-projects
    # Prompt: "Fix all critical CVEs and update lockfile"
    Container image scanning

    Scans Docker images with Trivy or Grype for OS and library CVEs before push to registry.

    trivy image --exit-code 1 \
    --severity HIGH,CRITICAL \
    myapp:latest
    grype myapp:latest -o json
    IAM & policy auditing

    Runs AWS IAM Access Analyzer, checks for overly permissive policies, unused roles, public S3 buckets, and open security groups.

    aws accessanalyzer list-findings
    aws s3api get-bucket-acl --bucket prod-assets
    aws ec2 describe-security-groups \
    --filters Name=ip-permission.cidr,Values='0.0.0.0/0'
    Compliance-as-code

    Writes OPA/Rego policies, AWS Config rules, or Azure Policy definitions. Checks Terraform plans against compliance rules before apply.

    # Checkov scans IaC for CIS benchmarks
    checkov -d . --framework terraform
    conftest test tfplan.json \
    --policy policy/
    Secrets Management Pattern Never hardcode credentials. Claude Code can write code that fetches secrets from AWS Secrets Manager, Vault, or Azure Key Vault at runtime, and can rotate secrets by calling the relevant CLI or API.

    06 — Monitoring & Incident Response

    Observability

    With Sentry, Datadog, or Grafana MCP servers connected, Claude Code can pull live error data, correlate with code, and push fixes — all without leaving the terminal.

    Sentry integration

    Connects via Sentry MCP. Claude reads the stack trace, finds the exact file/line in your local code, diagnoses the root cause, and applies a fix.

    "Check Sentry for any new CRITICAL errors in the last 2 hours. Show me the top 3 and fix the most frequent one."
    Datadog / Grafana

    Queries metrics and dashboards. Creates alert rules, writes runbooks, and interprets anomalies. Can also write custom Grafana dashboard JSON.

    "Our p99 latency spiked at 14:32 UTC. Query Datadog APM for the slowest traces during that window and identify the bottleneck."
    Log analysis

    Reads CloudWatch Logs, Loki, ELK, or raw log files. Extracts error patterns, builds queries, and creates alerting rules based on patterns found.

    "Query CloudWatch Logs for /ecs/api-service, last 30 min. Find all 5xx errors, group by endpoint, and add an alarm for any endpoint with >1% error rate."
    Incident runbooks

    Generates step-by-step runbooks for common failure scenarios based on your architecture. Stores them in your repo as markdown.

    "Create a runbook for 'RDS connection pool exhausted'. Include diagnosis steps, mitigation, and permanent fix recommendations."

    08 — Database Operations

    Postgres · MySQL · DynamoDB

    With a database MCP server, Claude Code reads your schema directly and writes accurate queries, migrations, and optimization recommendations without manual copy-paste.

    Schema-aware development

    Connects to your Postgres/MySQL via MCP, reads tables/indexes, then writes migration files, seed scripts, and optimized queries that actually match your schema.

    Scripts pg_dump/pg_restore workflows, sets up automated S3 backups, tests restore procedures.

    10 — Headless & Pipeline Automation

    CI Integration

    Run Claude Code non-interactively inside CI/CD pipelines for automated code review, security scanning, documentation generation, and deployment verification.

    Automated code review

    Add Claude Code to a GitHub Actions workflow triggered on pull requests. It reviews code, flags issues, and posts comments.

    # In GitHub Actions yaml
    - name: Claude Code Review
    run: |
    claude --print \
    "Review the diff in this PR for security issues, \
    bugs, and breaking changes. Output as markdown."
    env:
    ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
    Streaming JSON output

    For programmatic CI integration, use --output-format stream-json to parse Claude's actions as structured events.

    claude --output-format stream-json \
    --print \
    "Run tests and report failures" \
    | jq '.result'
    Post-deploy verification

    Runs smoke tests, checks health endpoints, verifies key flows after deployment and reports pass/fail.

    claude --print \
    "Hit the /health endpoint on $DEPLOY_URL. \
    Run the smoke test suite in tests/smoke/. \
    Output PASS or FAIL with details."

    11 — CLAUDE.md DevOps Blueprint

    Project Memory

    The .claude/CLAUDE.md file is loaded at the start of every session. It's your persistent memory — encode your stack, conventions, safety rules, and deployment procedures here. Commit it to git so the whole team shares context.

    # .claude/CLAUDE.md — example DevOps section
    ## Infrastructure
    - Cloud: AWS (eu-west-1 primary, us-east-1 DR)
    - IaC: Terraform 1.8, state in S3 + DynamoDB locking
    - Containers: EKS 1.30, Helm 3, Karpenter for node autoscaling
    - CI/CD: GitHub Actions, deploys via ArgoCD (GitOps)
    - Secrets: AWS Secrets Manager (never .env files in production)
    ## Deployment Rules
    - ALWAYS run terraform plan and show me output before apply
    - NEVER delete resources without explicit confirmation
    - Production deploys require a GitHub PR merged to main
    - Tag all resources with: team, environment, cost-center
    ## Security Checklist (run before every deploy)
    1. trivy image scan — no CRITICAL CVEs
    2. gitleaks detect — no secrets in diff
    3. checkov -d . — no HIGH severity IaC findings
    ## Environments
    - dev: auto-deploy from feature branches
    - staging: auto-deploy from main
    - production: manual approval gate in GitHub Actions

    Custom slash commands for DevOps

    Store in .claude/commands/ — invoke with /deploy, /security-scan, etc.

    # .claude/commands/security-scan.md

    12 — Enterprise Deployment Overview

    Production-Grade

    Enterprise cloud deployments are not just "bigger" versions of dev setups. They are architecturally different — designed around failure domains, blast radius, regulatory boundaries, and operational runbooks that must work at 3 AM without heroics.

    Scalability

    Horizontal auto-scaling, stateless services, sharding, caching layers. Zero performance degradation under 10× load.

    Responsiveness

    Sub-200ms p99 latency targets, CDN edge caching, async processing, circuit breakers, timeout policies.

    Security

    Zero-trust networking, mTLS service mesh, secrets rotation, RBAC least-privilege, runtime threat detection.

    Compliance

    SOC 2, ISO 27001, PCI-DSS, HIPAA, GDPR — automated controls, audit trails, policy-as-code enforcement.

    Observability

    Unified metrics + traces + logs (OpenTelemetry), SLO-based alerting, distributed tracing, anomaly detection.

    Reliability

    Multi-AZ, active-active multi-region, chaos engineering, automated DR drills, 99.99% SLA targets.

    13 — Scalability & Responsiveness

    Performance

    Horizontal scaling patterns

    Kubernetes HPA + KEDA

    HPA scales pods on CPU/memory. KEDA extends this to custom metrics: queue depth, request rate, Kafka lag, Prometheus metrics.

    # HPA on custom metric (RPS)
    apiVersion: autoscaling/v2
    kind: HorizontalPodAutoscaler
    spec:
    minReplicas: 3
    maxReplicas: 100
    metrics:
    - type: Pods
    pods:
    metric: {name: http_requests_per_second}
    target: {averageValue: "500"}
    Stateless service design

    Sessions in Redis/ElastiCache, files in S3/GCS, no local state on pods. Enables clean horizontal scale-out with zero sticky routing.

    # Claude Code prompt
    "Audit our Express app for any in-memory state or local file writes. Migrate sessions to Redis and uploads to S3. Show before/after diff."
    Database read scaling

    Read replicas for reporting queries, PgBouncer connection pooling, Redis caching layer for hot paths, CQRS pattern for write/read split.

    # Connection pool config
    pgbouncer:
    pool_mode: transaction
    max_client_conn: 1000
    default_pool_size: 20
    server_idle_timeout: 600

    Responsiveness: latency budget strategy

    LayerTarget latencyTechniqueClaude Code action
    CDN edge< 5msCloudFront / Fastly / Akamai static cachingGenerate cache-control headers, invalidation scripts
    API gateway< 10msResponse caching, request coalescingWrite Kong/AWS APIGW cache policies

    Async & event-driven patterns for high load

    Queue-based decoupling

    Move non-critical paths off the request thread. SQS, Kafka, RabbitMQ for order processing, notifications, batch jobs. Prevents cascading overload.

    "Refactor our /api/send-email endpoint to publish to SQS instead of calling SendGrid inline. Write the SQS consumer Lambda."
    Circuit breakers

    Prevent failure amplification. When a downstream service degrades, fail fast locally instead of queuing 10,000 timeout threads.

    "Add circuit breaker pattern to our payment service client using resilience4j. Trip after 50% failure in 10s window, recover after 30s."
    Graceful degradation

    Feature flags disable non-critical features under load. Stale cache serving when origin is slow. Partial response instead of all-or-nothing.

    "Add LaunchDarkly feature flags to our recommendation engine so we can serve cached results when ML service latency > 200ms."

    14 — Hardened Images & Runtime Security

    CIS Benchmarks

    Every layer of the runtime stack is an attack surface. Harden from the base OS image up through the container, pod, node, and cluster levels.

    Container image hardening checklist

    Distroless / scratch base

    Use gcr.io/distroless or cgr.dev/chainguard images. No shell, no package manager, no OS utilities — massively reduces attack surface.

    # Multi-stage: build on full image, ship distroless
    FROM node:20-alpine AS builder
    WORKDIR /app
    COPY . . && npm ci --only=production
    FROM gcr.io/distroless/nodejs20-debian12
    COPY --from=builder /app /app
    USER nonroot
    ENTRYPOINT ["node", "/app/server.js"]
    Non-root user + read-only FS

    Never run as root inside a container. Set runAsNonRoot: true, readOnlyRootFilesystem: true, and drop all Linux capabilities.

    # Kubernetes securityContext
    securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    readOnlyRootFilesystem: true
    allowPrivilegeEscalation: false
    capabilities:
    drop: ["ALL"]
    seccompProfile:
    type: RuntimeDefault
    Image signing & provenance

    Sign images with Cosign (Sigstore). Enforce signature verification at admission via Kyverno or OPA Gatekeeper. Maintain SBOM.

    # Sign after push
    cosign sign --key awskms:///alias/cosign \
    $ECR_URL/myapp:$SHA
    # Generate SBOM
    syft packages $ECR_URL/myapp:$SHA \
    -o spdx-json > sbom.json
    cosign attach sbom --sbom sbom.json \
    $ECR_URL/myapp:$SHA
    Runtime threat detection

    Falco monitors syscalls in real-time. Detects shell spawning in containers, unexpected network connections, privilege escalation attempts.

    # Falco rule example
    - rule: Shell in Container
    desc: A shell was spawned in a container
    condition: >
    container and proc.name in (shell_binaries)
    and not known_shell_spawn_binaries
    output: "Shell spawned (pod=%k8s.pod.name)"
    priority: WARNING
    Admission control

    Kyverno or OPA Gatekeeper enforce policies at deploy time: block privileged pods, require labels, mandate resource limits, block latest tag.

    # Kyverno: block :latest tag
    spec:
    rules:
    - name: require-image-tag
    match:
    resources: {kinds: [Pod]}
    validate:
    message: "Image tag 'latest' not allowed"
    pattern:
    spec.containers[].image: "!*:latest"
    Network policies

    Default-deny all ingress and egress. Explicitly allow only required pod-to-pod communication. Isolate namespaces by environment.

    # Default deny all
    apiVersion: networking.k8s.io/v1
    kind: NetworkPolicy
    metadata: {name: default-deny-all}
    spec:
    podSelector: {}
    policyTypes: [Ingress, Egress]

    15 — Enterprise Observability Stack

    OpenTelemetry · SLOs · Tracing

    True observability means you can answer any question about your system's behavior — without deploying new code. The three pillars (metrics, traces, logs) must be correlated by trace ID for this to work in practice.

    OpenTelemetry: unified instrumentation

    OTel SDK + Collector

    Instrument once with OTel SDK. Route signals through the OTel Collector to any backend (Datadog, Grafana, Honeycomb, Jaeger) without code changes.

    # OTel Collector config
    receivers:
    otlp: {protocols: {grpc: {}, http: {}}}
    processors:
    batch: {timeout: 10s, send_batch_size: 1000}
    resource: {attributes: [{action: insert,
    key: env, value: production}]}
    exporters:
    otlphttp/datadog: {endpoint: $DD_OTLP_ENDPOINT}
    prometheusremotewrite: {endpoint: $GRAFANA_URL}
    SLO-based alerting

    Define Service Level Objectives. Alert on error budget burn rate, not raw error count. A 5% error rate matters very differently at 3 AM vs during a load spike.

    # SLO definition (Sloth format)
    objectives:
    - name: api-availability
    target: 0.999 # 99.9%
    window: 30d
    indicator:
    events:
    error_query: sum(rate(http_requests_total{status=~"5.."}[1m]))
    total_query: sum(rate(http_requests_total[1m]))
    Distributed tracing

    Propagate trace context (W3C TraceContext) across all services. A single trace ID links API gateway → service A → service B → database query → cache hit.

    "Add OpenTelemetry auto-instrumentation to our Node.js services. Configure trace context propagation between all services. Export to Tempo via OTel Collector."
    Structured logging

    JSON logs with consistent fields: trace_id, span_id, service, environment, user_id. Enables correlated log querying across services in one Loki/CloudWatch query.

    {
    "level": "error",
    "trace_id": "4bf92f3577b34da6",
    "service": "payment-svc",
    "env": "production",
    "msg": "charge failed",
    "user_id": "u_12345",
    "duration_ms": 2341
    }

    16 — Compliance & Governance

    SOC 2 · PCI · HIPAA · GDPR

    Compliance is not a one-time audit — it is a continuous control plane. Use policy-as-code to make compliance checks automatic, auditable, and reviewable in pull requests.

    Policy-as-code (OPA)

    Write Rego policies that enforce compliance requirements at Terraform plan time, Kubernetes admission, and CI/CD gate. Violations block deployment.

    # OPA/Rego: enforce encryption at rest
    deny[msg] {
    resource := input.resource.aws_s3_bucket[name]
    not resource.server_side_encryption_configuration
    msg := sprintf("S3 bucket '%v' must have SSE enabled", [name])
    }
    Immutable audit trails

    All infrastructure changes flow through Terraform + Git. Every API call logged to CloudTrail/Audit Logs. Logs shipped to immutable S3 (Object Lock) or Worm storage.

    # Enable CloudTrail with log file validation
    aws cloudtrail create-trail \
    --name enterprise-audit-trail \
    --s3-bucket-name audit-logs-immutable \
    --enable-log-file-validation \
    --is-multi-region-trail
    Data residency & GDPR

    Tag resources with data classification. Enforce regional boundaries via SCP/Azure Policy. Auto-detect PII in logs using DLP scanning and mask before storage.

    "Scan our application logs for PII patterns (email, phone, SSN, credit card). Add log masking middleware. Generate a GDPR data flow map from our codebase."
    Automated evidence collection

    For SOC 2 audits, automate evidence gathering: access review exports, change management logs, vulnerability scan reports, backup test results.

    "Generate a SOC 2 Type II evidence package: export last 90 days of CloudTrail, access review CSV from IAM, last 5 Trivy scan reports, and backup restore test logs."

    17 — Multi-Cloud Strategy

    AWS · GCP · Azure

    Multi-cloud is not just "run on two clouds." Done well, it means cloud-agnostic abstraction layers, portable workloads, and the operational discipline to manage increased complexity in exchange for resilience and vendor independence.

    Multi-cloud deployment models

    Active-Active

    Traffic split across two clouds simultaneously. Global load balancer (Cloudflare / AWS Global Accelerator) routes to nearest healthy region. Zero-downtime cloud failover.

    # Cloudflare load balancer config
    origins:
    - name: aws-us-east-1
    address: api.aws.example.com
    weight: 50
    - name: gcp-us-central1
    address: api.gcp.example.com
    weight: 50
    health_check: {path: /health, interval: 10s}
    Active-Passive DR

    Primary on one cloud, warm standby on another. RTO < 15 minutes. Data replicated continuously. Failover triggered by health check failure or manual operator action.

    "Set up a DR configuration where AWS us-east-1 is primary and GCP us-central1 is warm standby. Write Terraform for both, with cross-cloud PostgreSQL replication via pglogical."
    Best-of-breed selection

    Use each cloud for what it does best: GCP BigQuery for analytics, AWS SageMaker for ML, Azure AD for enterprise identity, Cloudflare for edge.

    # Example cross-cloud service mesh
    AWS EKS → core application
    GCP BigQuery → analytics pipeline
    Azure AD → enterprise SSO / OIDC
    Cloudflare → CDN + WAF + Zero Trust

    Multi-cloud data synchronization

    Database replication

    Cross-cloud Postgres streaming replication (pglogical), CockroachDB multi-region, or PlanetScale for MySQL. Target < 5s replication lag for DR.

    Event streaming

    Kafka with MirrorMaker 2 or Confluent Replicator for cross-cloud event sync. Enables real-time data parity between active-active deployments.

    Object storage sync

    Rclone scheduled sync or AWS DataSync for S3→GCS→Azure Blob. Critical for backup redundancy and regulatory data residency requirements.

    18 — Well-Architected Cloud-Native K8s Deployment

    5 Pillars

    The AWS, GCP, and Azure Well-Architected Frameworks share five core pillars. Applied to Kubernetes deployments, each pillar translates to concrete configurations and Claude Code prompts.

    Pillar 1 — Operational Excellence

    GitOps with ArgoCD / Flux

    The cluster state is always derived from Git. No manual kubectl applies in production. Drift is detected and auto-corrected. Full audit trail of every change.

    # ArgoCD Application manifest
    spec:
    source:
    repoURL: https://github.com/org/k8s-config
    targetRevision: main
    path: apps/production
    syncPolicy:
    automated: {prune: true, selfHeal: true}
    Progressive delivery

    Blue/green, canary, and feature-flag-gated releases using Argo Rollouts or Flagger. Automated rollback triggered when error rate SLO is breached.

    # Argo Rollouts canary
    strategy:
    canary:
    steps:
    - setWeight: 5
    - pause: {duration: 10m}
    - setWeight: 20
    - analysis: {templates: [{name: success-rate}]}
    - setWeight: 100

    Pillar 2 — Security

    Zero-trust service mesh

    Istio with STRICT mTLS mode. Every service-to-service call is mutually authenticated and encrypted. AuthorizationPolicies define explicit allow-lists.

    apiVersion: security.istio.io/v1beta1
    kind: PeerAuthentication
    metadata: {name: default, namespace: production}
    spec:
    mtls: {mode: STRICT}
    Workload identity (IRSA/WI)

    Pods never hold long-lived cloud credentials. AWS IRSA / GKE Workload Identity / AKS Pod Identity bind K8s service accounts to cloud IAM roles.

    "Configure IRSA for our payment-service pod to access DynamoDB and SecretsManager using least-privilege IAM policy. No access keys."

    Pillar 3 — Reliability

    Pod disruption budgets

    Guarantee minimum available replicas during node drain, cluster upgrades, or rolling updates. Prevent accidental full-service outage during maintenance.

    apiVersion: policy/v1
    kind: PodDisruptionBudget
    spec:
    minAvailable: 2 # or 80%
    selector:
    matchLabels: {app: api-service}
    Multi-AZ pod spreading

    Topology spread constraints ensure pods are distributed across AZs. Node affinity rules prevent co-location of all replicas on the same node.

    topologySpreadConstraints:
    - maxSkew: 1
    topologyKey: topology.kubernetes.io/zone
    whenUnsatisfiable: DoNotSchedule
    labelSelector:
    matchLabels: {app: api}
    Chaos engineering

    LitmusChaos or Chaos Monkey validates resilience. Kill random pods, inject network latency, drain nodes — verify your SLOs hold before an incident does it for you.

    "Write a LitmusChaos experiment that kills 30% of api-service pods every 5 minutes for 1 hour. Alert me if error rate breaches 0.1%."

    Pillar 4 — Performance Efficiency

    Right-sized resource requests

    VPA (Vertical Pod Autoscaler) in recommendation mode suggests CPU/memory based on real usage. Avoid overprovisioning (wasted cost) and underprovisioning (OOMKills).

    # Get VPA recommendations
    kubectl get vpa api-service -o yaml
    # Claude reads + applies recommendations
    "Apply VPA recommendations for all deployments in the production namespace. Update resource requests/limits in values.yaml."
    Karpenter node provisioning

    Karpenter provisions the right instance type for each workload's actual resource needs — not a fixed node pool. Spot instances for stateless workloads.

    spec:
    disruption:
    consolidationPolicy: WhenUnderutilized
    requirements:
    - key: karpenter.sh/capacity-type
    operator: In
    values: [spot, on-demand]
